# LL(1) 语法分析器项目详细设计文档

## 1. 项目作用

本项目是一个基于 Rust 实现的 LL(1) 语法分析器，主要用于：

- 编译原理课程作业
- LL(1) 分析流程演示
- 基础语法分析实验验证

它的核心目标不是做一个完整的编译器前端，而是把 LL(1) 语法分析这条主链路做成一个可执行、可测试、可展示的完整程序。

程序能够完成以下工作：

- 读取文法和输入串
- 计算 `FIRST` 集
- 计算 `FOLLOW` 集
- 构造 LL(1) 预测分析表
- 对输入串执行预测分析
- 输出终端摘要
- 生成 Markdown 分析报告

## 2. 项目定位

本项目采用“最小完整实现”的设计思路。

具体取舍如下：

- 只支持 LL(1) 分析
- 不自动消除左递归
- 不自动提取左公因子
- 不做图形界面
- 不做语法树可视化

原因很简单：课程项目最重要的是“逻辑完整、实现清楚、结果可展示”。因此本项目把精力集中在 LL(1) 主流程上，而不是扩展过多附加功能。

## 3. 总体工作流程

程序整体流程如下：

```txt
输入文件
  -> 文法与输入串解析
  -> FIRST 集计算
  -> FOLLOW 集计算
  -> 预测分析表构造
  -> 输入串逐条分析
  -> 终端摘要输出
  -> Markdown 报告生成
```

这个流程完全对应 LL(1) 分析的标准理论步骤，因此既便于实现，也便于讲解。

## 4. 输入设计

项目使用固定格式文本输入，而不是交互式输入。这样做的优点是：

- 便于保存和复现
- 便于自动化测试
- 便于批量分析多条输入串
- 便于课程验收时直接展示

示例：

```txt
%start Expr
%token id + * ( )
%%
Expr -> Term ExprP
ExprP -> + Term ExprP | ε
Term -> Factor TermP
TermP -> * Factor TermP | ε
Factor -> ( Expr ) | id
%%
id + id * id
( id + id ) * id
id + + id
```

含义如下：

- `%start`：定义开始符号
- `%token`：定义终结符
- 第一段 `%%` 到第二段 `%%`：产生式定义
- 第二段 `%%` 之后：每行一条输入串

## 5. 核心模块设计

### 5.1 `grammar`

负责定义项目中的核心数据结构，包括：

- `Symbol`
- `Production`
- `Grammar`
- `ParseStep`
- `ParseResult`
- `AnalysisReport`

这是整个项目的数据基础层，后续所有模块都依赖这些结构。

### 5.2 `loader`

负责输入文件解析。

主要逻辑：

1. 按 `%%` 分段
2. 解析 `%start` 和 `%token`
3. 第一轮收集所有非终结符
4. 第二轮构建产生式，区分终结符/非终结符/空串
5. 解析输入串
6. 做基础合法性检查

这里采用“两轮处理”的原因是：如果只扫描一遍，右部里的非终结符很容易被误判成终结符。

### 5.3 `first_follow`

负责计算 `FIRST` 集和 `FOLLOW` 集。

实现方式是固定点迭代。

这样做的好处是：

- 逻辑直接对应理论定义
- 代码结构清楚
- 便于测试验证

### 5.4 `parse_table`

负责构造 LL(1) 预测分析表。

主要功能：

- 按 `FIRST` 和 `FOLLOW` 计算表项
- 检测表项冲突
- 对缺失 `FIRST/FOLLOW` 的情况给出显式错误

这一层的关键作用是把“文法能否作为 LL(1) 文法使用”判断清楚。

### 5.5 `analyzer`

负责基于预测分析表执行栈驱动分析。

分析过程记录为一系列 `ParseStep`，包括：

- 当前步号
- 当前分析栈
- 当前剩余输入
- 当前动作
- 是否出错

这样既能支撑终端展示，也能支撑报告输出。

### 5.6 `report`

负责把最终结果渲染成 Markdown 报告。

报告内容包括：

- 原始文法
- `FIRST` 集
- `FOLLOW` 集
- 预测分析表
- 每条输入串的分析过程

为了保证报告结构稳定，项目还对动态内容做了 Markdown 转义和单行化处理。

### 5.7 `main`

负责命令行入口和整体调度。

主要职责：

- 读取参数
- 调用各模块
- 输出终端摘要
- 写出分析报告

主程序只负责“串流程”，不承担算法细节。

## 6. 关键数据结构说明

### 6.1 `Symbol`

符号分为四类：

- `Terminal(String)`
- `NonTerminal(String)`
- `Epsilon`
- `EndMarker`

这样设计的意义在于：算法阶段不再需要通过字符串猜测符号种类。

### 6.2 `Production`

每条产生式包含：

- 唯一编号 `id`
- 左部 `left`
- 右部 `right`

产生式编号非常重要，因为：

- 预测分析表项最终指向产生式编号
- 分析步骤里也要引用具体产生式

### 6.3 `Grammar`

`Grammar` 结构中保存：

- 开始符号
- 终结符集合
- 非终结符集合
- 产生式列表
- 按左部分组的产生式映射

这让后续查找和分析都更方便。

### 6.4 `ParseStep` 与 `ParseResult`

`ParseStep` 表示一步预测分析动作，`ParseResult` 表示一整条输入串的最终分析结果。

这样设计的好处是：

- 可以直接生成分析过程表格
- 可以用于测试断言
- 可以在出错时保留完整上下文

### 6.5 `AnalysisReport`

`AnalysisReport` 是最终报告的数据汇总结构，用来统一承载：

- 文法
- 集合
- 分析表
- 分析结果

报告模块只负责“渲染”，不负责“计算”。

## 7. FIRST 集实现原理

`FIRST` 集表示：一个符号串最先可能出现哪些终结符。

实现方法：

1. 为每个非终结符初始化空集合
2. 遍历所有产生式
3. 调用 `first_of_sequence` 计算右部的 `FIRST`
4. 把结果加入左部
5. 重复直到集合不再变化

其中 `first_of_sequence` 采用统一契约：

- 返回“不含 `ε` 的 FIRST 集”
- 再返回一个 `nullable` 标记表示该串是否可空

这个契约同时服务于 `FIRST` 集计算、`FOLLOW` 集计算和预测分析表构造。

## 8. FOLLOW 集实现原理

`FOLLOW` 集表示：某个非终结符后面可能出现哪些终结符。

实现逻辑：

1. 开始符号先加入 `$`
2. 扫描每条产生式右部
3. 若当前位置是非终结符，则分析它后面的符号串
4. 将后缀 `FIRST` 去掉 `ε` 后并入当前非终结符的 `FOLLOW`
5. 若后缀可空，则把左部的 `FOLLOW` 传播给它
6. 不断迭代直到稳定

这种写法直接对应教材规则，便于推导和检查。

## 9. 预测分析表构造原理

预测分析表的本质是一个二维映射：

```txt
M[非终结符, 当前输入终结符] = 产生式编号
```

构造原则：

- 对产生式 `A -> α`，将 `FIRST(α)` 中的终结符写入对应表项
- 如果 `α` 可空，则把 `FOLLOW(A)` 中的符号也写入对应表项

项目在这一阶段显式处理了三类错误：

- `Conflict`：同一表项被不同产生式重复填入
- `MissingFirstSet`：缺失非终结符的 `FIRST`
- `MissingFollowSet`：缺失非终结符的 `FOLLOW`

这样可以把构表失败明确区分为“文法冲突”还是“输入数据不完整”。

## 10. 预测分析实现原理

预测分析使用“栈 + 输入指针 + 分析表”的经典方法。

基本过程：

1. 栈初始化为 `$` 和开始符号
2. 输入尾部补 `$`
3. 取栈顶和当前输入符号
4. 按情况处理：

- 终结符相同：匹配成功
- 栈顶为非终结符：查表并展开
- 栈顶为 `ε`：跳过
- 查表失败或终结符不匹配：返回错误

本项目还额外处理了一个工程问题：

- 如果分析表中的产生式编号在文法里找不到，不再 panic，而是返回一条可诊断的错误结果

这样保证分析器整体保持“可恢复”风格。

## 11. 报告生成原理

报告使用 Markdown 格式，原因是：

- 易读
- 易保存
- 易提交
- 易展示

报告内容分为几块：

- 原始文法
- `FIRST` 集
- `FOLLOW` 集
- 预测分析表
- 每条输入串的逐步分析结果

为避免用户输入破坏 Markdown 结构，项目会对动态内容进行清洗：

- `|` 转义
- 反引号转义
- 换行和回车单行化
- 多余空白压缩

这保证了报告在面对特殊输入时依然结构正确。

## 12. 主流程设计逻辑

主程序 `run()` 的执行顺序如下：

1. 解析参数
2. 读取输入文件
3. 计算 `FIRST`
4. 计算 `FOLLOW`
5. 构造分析表
6. 逐条分析输入串
7. 输出终端摘要
8. 生成 Markdown 报告

这一设计的优点是：

- 流程直观
- 每一步都可单独测试
- 出错位置清楚

这些测试分别覆盖：

- 输入解析

## 14. 总结

这个项目本质上是一个“面向课程作业的 LL(1) 语法分析器工程实现”。

它的价值主要在于：

- 把教材中的分析流程完整落地
- 保持实现简单而完整
- 提供可验证、可展示的结果
- 保留清晰的工程结构，便于扩展和说明

对于课程项目来说，这种设计比功能堆叠更重要，因为它真正把“语法分析原理”转换成了“可运行、可测试、可讲解的程序”。
