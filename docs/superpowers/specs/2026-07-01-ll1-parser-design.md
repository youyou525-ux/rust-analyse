# Rust 语言 LL(1) 语法分析器设计说明

## 1. 项目目标

本项目要实现一个基于 Rust 语言的 LL(1) 语法分析器，定位为课程作业型、可直接演示的最小完整实现。

程序从文件中读取一组已经整理好的 LL(1) 文法和若干输入串，完成以下工作：

- 计算 `FIRST` 集
- 计算 `FOLLOW` 集
- 构造 LL(1) 预测分析表
- 对输入串进行预测分析
- 输出终端摘要结果
- 生成可保存的详细分析报告

整个项目优先强调以下目标：

- 实现尽量简单，保证能落地
- 使用方式尽量简单，方便演示
- 分析流程尽量可见，便于课程展示
- 结构尽量清晰，便于编写基础测试

## 2. 范围说明

### 2.1 范围内

- 从文本文件读取文法和输入串
- 显式区分终结符与非终结符
- 计算 `FIRST` 集
- 计算 `FOLLOW` 集
- 构造 LL(1) 预测分析表
- 检查给定文法是否存在 LL(1) 表冲突
- 对同一文件中的多条输入串逐条分析
- 记录每一步分析过程
- 输出简洁的终端结果
- 生成 Markdown 详细报告
- 编写少量但清晰的 Rust 单元测试与集成测试

### 2.2 范围外

- 一般上下文无关文法分析
- 自动消除左递归
- 自动提取左公因子
- 语法树图形化展示
- 图形界面或网页界面
- 面向大规模文法的性能优化
- 错误恢复策略，例如同步恢复、恐慌模式恢复

## 3. 总体设计

系统采用单向流水线结构：

`输入文件 -> 内部文法模型 -> FIRST/FOLLOW -> 预测分析表 -> 输入串分析 -> 终端摘要/报告输出`

这样设计的原因很直接：

- 实现路径短，复杂度低
- 每个阶段职责清楚，便于调试
- 演示时可以按步骤展开说明
- 测试时可以分模块验证

## 4. 模块划分

### 4.1 `grammar`

负责定义核心数据结构，包括：

- 符号类型
- 产生式
- 文法对象

该模块只关心“文法在程序里如何表示”，不负责算法计算。

### 4.2 `loader`

负责从输入文件中读取并解析：

- 开始符号
- 终结符声明
- 产生式列表
- 输入串列表

该模块只做格式解析与基础合法性检查，不做 LL(1) 自动改写。

### 4.3 `first_follow`

负责计算：

- 每个非终结符的 `FIRST` 集
- 每个非终结符的 `FOLLOW` 集

该模块不负责分析表构造，也不负责输入串分析。

### 4.4 `parse_table`

负责依据文法和 `FIRST/FOLLOW` 集构造 LL(1) 预测分析表。

如果某个表项被不同产生式重复填入，则说明文法存在冲突，程序应明确报告该问题。

### 4.5 `analyzer`

负责执行预测分析过程。

该模块维护：

- 分析栈
- 当前输入位置
- 当前动作
- 分析步骤记录

### 4.6 `report`

负责统一输出结果，包括：

- 终端摘要
- Markdown 报告

算法模块只提供结构化数据，不直接负责拼展示文本，这样更清晰。

### 4.7 `cli`

负责处理命令行参数，例如：

- 输入文件路径
- 输出目录
- 是否生成详细报告

## 5. 输入格式设计

为保证使用简单，程序只接收一种固定格式的文本文件。推荐格式如下：

```txt
%start Expr
%token id + * ( )
%%
Expr -> Term ExprP
ExprP -> + Term ExprP | ε
Term -> Factor TermP
TermP -> * Factor TermP | ε
Factor -> ( Expr ) | id
%%
id + id * id
( id + id ) * id
id + + id
```

### 5.1 格式含义

- `%start` 用于声明开始符号
- `%token` 用于显式声明终结符
- 第一段 `%%` 与第二段 `%%` 之间是产生式定义
- 第二段 `%%` 之后每一行表示一条待分析输入串

### 5.2 选择该格式的原因

- 文本直观，适合课程作业展示
- 解析规则稳定，实现简单
- 不需要程序猜测哪些符号是终结符
- 一次运行即可处理多条输入串
- 使用方式统一，便于测试

## 6. 核心数据结构设计

### 6.1 `Symbol`

建议使用枚举形式表示符号，逻辑上包含以下几类：

- `Terminal(String)`：终结符
- `NonTerminal(String)`：非终结符
- `Epsilon`：空串
- `EndMarker`：输入结束符 `$`

### 6.2 `Production`

每条产生式至少包含：

- 唯一编号
- 左部非终结符
- 右部符号序列

产生式编号的作用是让分析步骤和报告引用更直接，例如“第 3 条产生式”。

### 6.3 `Grammar`

文法对象至少包含：

- 开始符号
- 终结符集合
- 非终结符集合
- 产生式列表
- 按左部分组的产生式映射

### 6.4 `ParseStep`

表示预测分析中的一步，建议至少包含：

- 步号
- 当前分析栈快照
- 当前剩余输入串
- 当前执行动作
- 使用的产生式编号或匹配符号
- 本步是否出错

### 6.5 `AnalysisReport`

用于承载一次完整运行中需要展示的数据，建议包含：

- 原始文法
- `FIRST` 集
- `FOLLOW` 集
- 预测分析表
- 每条输入串的分析结果

## 7. 算法流程设计

### 7.1 输入文件解析

程序首先将输入文件解析为：

- 一个 `Grammar`
- 一个输入串列表

这一阶段主要检查：

- 是否存在 `%start`
- 产生式格式是否正确
- 符号是否已声明
- 开始符号是否合理

如果这里失败，程序直接终止，并输出明确错误信息。

### 7.2 `FIRST` 集计算

采用迭代法计算，直到集合不再变化。

基本规则如下：

- 终结符的 `FIRST` 集是它本身
- 遇到产生式右部时，从左到右依次传播
- 只有当前面的符号都可推出空串时，才继续向后看
- 如果整条右部都可推出空串，则将 `ε` 加入左部的 `FIRST` 集

### 7.3 `FOLLOW` 集计算

同样采用迭代法计算。

基本规则如下：

- 开始符号的 `FOLLOW` 集包含 `$`
- 对于 `A -> αBβ`，将 `FIRST(β)` 中除 `ε` 外的符号加入 `FOLLOW(B)`
- 如果 `β` 可推出 `ε`，则将 `FOLLOW(A)` 加入 `FOLLOW(B)`

### 7.4 预测分析表构造

对每条产生式 `A -> α`：

- 将 `FIRST(α)` 中除 `ε` 以外的终结符对应表项填为该产生式
- 如果 `FIRST(α)` 中包含 `ε`，则将 `FOLLOW(A)` 中的符号对应表项填为该产生式

如果某个表项需要被不同产生式重复填入，则说明该文法不是可直接使用的 LL(1) 文法，程序应输出冲突信息并停止后续输入串分析。

这一处理方式符合“最简单可实现”的目标：程序只验证和展示，不自动修正文法。

### 7.5 预测分析过程

对每条输入串执行以下过程：

1. 在输入尾部补上 `$`
2. 初始化分析栈为 `$` 和开始符号
3. 循环执行，直到接受或出错

具体情况如下：

- 若栈顶与当前输入符号均为相同终结符，则匹配成功，弹栈并前进
- 若栈顶为非终结符，则查询预测分析表并按产生式展开
- 若栈顶为 `ε`，则直接弹出
- 若查表失败或终结符不匹配，则判定当前输入串分析失败

每一步都要记录为一条 `ParseStep`，以便终端和报告统一展示。

这种“栈 + 输入指针 + 表驱动”的分析方式也与参考仓库 `Herdeny/LL1_Remake` 的教学展示思路一致，便于实现与答辩说明。

## 8. 错误处理设计

### 8.1 输入错误

典型情况包括：

- 输入文件缺少必要段落
- 产生式格式错误
- 使用了未声明符号
- 开始符号无效

处理方式：

- 直接停止程序
- 输出明确错误原因
- 不进入后续分析阶段

### 8.2 预测分析表构造错误

典型情况包括：

- 同一表项出现多个候选产生式
- 给定文法无法直接构成 LL(1) 预测分析表

处理方式：

- 不视为程序崩溃
- 输出原始文法
- 输出已计算的 `FIRST/FOLLOW`
- 输出冲突位置和冲突原因
- 停止输入串分析

### 8.3 输入串分析错误

典型情况包括：

- 查表失败
- 终结符不匹配
- 输入未消费完但栈已结束

处理方式：

- 只判定当前输入串失败
- 继续处理同文件中的其他输入串
- 在步骤记录中明确标出失败位置与原因

## 9. 输出设计

### 9.1 终端输出

终端输出应尽量简洁，只展示关键结果，例如：

- 文法是否成功构造预测分析表
- 每条输入串是否接受
- 报告文件输出路径

### 9.2 Markdown 报告

为保证流程可见，程序应生成结构固定的 Markdown 报告，建议包含以下章节：

1. 实验概览
2. 原始文法
3. `FIRST` 集
4. `FOLLOW` 集
5. 预测分析表
6. 各输入串的分析过程
7. 错误总结（如有）

其中每条输入串的分析过程应至少包含：

- 输入串内容
- 最终结论：接受或拒绝
- 分析步骤表：步号、分析栈、剩余输入、动作

## 10. 测试方案

### 10.1 单元测试

建议至少覆盖以下模块：

#### `loader`

验证：

- 合法输入能被正确解析
- 非法输入能返回明确错误

#### `first_follow`

验证：

- 经典表达式文法的 `FIRST` 集是否正确
- 经典表达式文法的 `FOLLOW` 集是否正确

#### `parse_table`

验证：

- 关键表项是否构造正确
- 冲突文法是否能被识别

#### `analyzer`

验证：

- 合法输入串能被接受
- 非法输入串会被拒绝
- 含空产生式的情况能被正确处理

### 10.2 集成测试

通过命令行运行完整样例文件，验证：

- 程序退出状态是否正确
- 终端输出是否包含关键摘要
- 报告文件是否成功生成

### 10.3 建议样例

建议准备以下几类样例：

- 经典表达式文法
- 含空产生式的简单文法
- 故意构造冲突的文法

这些样例既可以用于测试，也可以直接用于课堂展示。

## 11. 实现约束

项目实现应遵循以下原则：

- 优先清晰，不做过早优化
- 优先简单稳定，不追求过度抽象
- 算法与展示分离
- 输入与输出格式固定，减少使用复杂度
- 功能宁可少而完整，不要多而杂

对课程作业而言，可解释性、正确性和可演示性比性能更重要。

## 12. 风险与应对

### 风险一：`FIRST` 和 `FOLLOW` 计算容易出错

应对方式：

- 先用经典文法做单元测试
- 将集合计算与 CLI 分开测试

### 风险二：输出过程过多导致展示混乱

应对方式：

- 终端只展示摘要
- 详细过程全部放入 Markdown 报告

### 风险三：给定文法并不是 LL(1)

应对方式：

- 将其视为正常分析结果，而不是程序异常
- 输出冲突信息，说明不能继续构造有效分析表

## 13. 最终建议

本项目应采用“小而完整”的 Rust LL(1) 语法分析器设计：

- 输入使用固定文本格式
- 文法由用户预先整理为适合 LL(1) 分析的形式
- 程序重点完成 `FIRST/FOLLOW`、分析表构造和预测分析过程展示
- 输出同时兼顾终端摘要与详细报告
- 通过少量但关键的测试保证正确性

与参考仓库 `Herdeny/LL1_Remake` 相比，本设计保留了相同的核心教学路线：

- 已准备好的文法输入
- `FIRST` 与 `FOLLOW` 计算
- 预测分析表构造
- 栈驱动分析过程

不同点在于，本项目将使用 Rust 实现，并采用更规范的文件输入和报告输出方式，以保证在保持简单的前提下，更容易验证和展示。
